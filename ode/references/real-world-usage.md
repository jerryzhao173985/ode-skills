# Real-world ODE usage βÄî who runs it, how they integrate it

> **Framing (load-bearing).** Everything here is WEB-SOURCED ECOSYSTEM / USAGE signal (each claim carries a source + an approximate signal + a year), **not** API truth. It establishes *who* depends on ODE and *how* they wire it βÄî useful because ODE's home is `bitbucket.org/odedevs/ode`, so GitHub-star counts undercount it; judge it by its downstream consumers. **Cross-check every symbol against `include/ode/*.h`; this file never overrides a header.** API detail lives in the core references; this file is orientation only.

## Production consumers worth learning from
- **Webots** (`github.com/cyberbotics/webots`, ~4.4kβ‰Ö, active 2026) βÄî flagship robotics simulator built **on** ODE as its physics backend. Vendors a *customized in-tree ODE fork* (bundling OPCODE + libccd) and feeds density / CoM / inertia / damping / restitution straight into ODE bodies and contacts, resolving contact materials *per pair* at collision time. Ecosystem-validated starting defaults it exposes (`cyberbotics.com/doc/reference/contactproperties`, 2025): timestep **8βÄ™16 ms**, global ERP 0.2 / CFM 1e-5, per-contact coulombFriction 1 / bounce 0.5 / softERP 0.2 / softCFM 0.001 / maxContactJoints 10. Treat as tuning *starting points*, not API rules.
- **OpenRAVE** (`github.com/rdiankov/openrave`, ~0.8kβ‰Ö) βÄî uses ODE as its **default collision checker** and an optional physics engine; the canonical proof that ODE's collision half runs **standalone**, decoupled from its dynamics solver. "ODE for collision only" is a legitimate real-world mode.
- **CoppeliaSim / V-REP** (`coppeliarobotics.com`, v4.x 2025) βÄî ODE is **1 of 5 swappable engines**; the A/B-across-engines reproducibility workflow. Documents ODE default-config **non-determinism** (a different result per run unless configured) and ~**64 contacts/pair**.
- **lpzrobots** (`github.com/georgmartius/lpzrobots`, academic) βÄî vendors a **renamed double-precision build (`ode-dbl`)** to avoid clashing with a packaged single-precision ODE (a real precision/packaging gotcha); uses a "control interval" (controller every N physics steps) and an ODE angular motor + PID servo with command in [-1, 1].
- **Gazebo Classic** (`gazebosim/gazebo-classic`, **EOL 2025**) βÄî historically the largest ODE deployment (DARPA Robotics Challenge, early ROS); **legacy** βÄî beware its stale solver tuning. Modern Gazebo (`gz-physics`) keeps ODE only as a collision backend behind DART.

## Integration patterns the single-program tutorial omits
Every ODE symbol named below is cross-checked against `include/ode/*.h`; the patterns themselves are ecosystem signal.
- **Fixed-step physics + control decimation.** Step a small *fixed* dt; call the policy/controller every N steps (lpzrobots' "control interval"). Never vary dt per frame (`references/foundations/stepping-and-stability.md`).
- **Reset / serialize by hand.** ODE ships `dWorldExportDIF` (a debug text dump) but **no importer / checkpoint** βÄî verified: `include/ode/export-dif.h:33` declares only `dWorldExportDIF(dWorldID, FILE*, const char*)`, no `dWorldImportDIF`. An `env.reset()` is hand-rolled: capture per-body `dBodyGetPosition` / `dBodyGetQuaternion` / `dBodyGetLinearVel` / `dBodyGetAngularVel` (+ joint state) into a struct, restore with the matching setters, and zero accumulated forces.
- **Vectorized / batched envs.** N independent `dWorldID` + `dSpaceID` stepped across a thread pool (EnvPool-style) βÄî embarrassingly parallel, and *distinct* from ODE's per-island threading inside one world (`references/threading.md`).
- **Headless throughput.** Drop the renderer, pin dt, prefer `dWorldQuickStep` with a raised `dWorldSetQuickStepNumIterations` (default 20 βÜ£ ~50βÄ™100), enable auto-disable, and avoid per-step `malloc` via the step-memory APIs (`dWorldSetStepMemoryReservationPolicy` `objects.h:293`, `dWorldSetStepMemoryManager` `objects.h:348`).
- **RL-ergonomics reality.** ODE has no Gym wrapper, no official Python API, no URDF/MJCF import, and no built-in sensors βÄî its audience is the integrator who builds the env / sensor / serialization scaffolding on the raw C API, and it most often appears as a *backend* inside Gazebo/Webots rather than used directly (multiple 2024 RL-simulator surveys).

## Obtain matrix (2024βÄ™2026, ecosystem signal)
- **macOS:** `brew install ode` (0.16.6, autotools bottle, **no drawstuff**). **Debian/Ubuntu:** `apt install libode-dev` (double on 64-bit, **no drawstuff**). **From source (any OS):** `git clone https://bitbucket.org/odedevs/ode && git checkout 0.16.6 && cmake -B build && cmake --build build`. **C/C++:** Conan `ode/0.16.x`, vcpkg `ode`. **Python:** `pip install ode-python` (ctypes) or conda `pyode`. **Go:** `ianremmler/ode`. **Rust:** `ode-rs`. **Web:** Emscripten/WASM ports.
- **drawstuff ships with NEITHER Homebrew NOR apt** βÄî it builds only inside the demos block (CMake `ODE_WITH_DEMOS`, needs OpenGL/GLUT + X11). This **generalizes** the skill's "default to headless" guidance (`references/rendering-and-headless.md`) to *all* package-manager installs, not just Homebrew.
- **Precision = double is the cross-toolchain default** (CMake / apt / Conan / Go / Rust all default to double on 64-bit). Flag discovery: `make install` ships both `ode-config` and `ode.pc`; prefer `pkg-config --cflags --libs ode`, with `ode-config` as the fallback. (Stale-API and `dSINGLE`/`dDOUBLE` traps are owned by `references/version-and-changelog.md` and `references/building-and-running.md`.)

## Where ODE sits among engines (2025 status βÄî trajectory only)
The decision **table** lives in `references/foundations/scope-and-limits.md` ¬ß1; this is status signal for trajectory: **MuJoCo** (Apache-2.0, the robotics/RL standard; GPU via MJX) ¬Ζ **Bullet/PyBullet** (maintenance mode since ~2022) ¬Ζ **Jolt** (MIT, very active; default physics in Godot 4.4+) ¬Ζ **PhysX 5** (GPU-first, NVIDIA Isaac) ¬Ζ **DART** (Featherstone, white-box research) ¬Ζ **Project Chrono** (multiphysics / FEA). ODE persists where a *lightweight, embeddable C dynamics + collision library you fully control* beats a heavyweight engine βÄî exactly the OpenRAVE/Webots "ODE as a component" pattern above.
